Monolito vs Microservicios

Un monolito es una sola aplicación gigante. Los microservicios son procesos pequeños e independientes. Si falla uno (ej: Reviews), el resto (productos, carrito) sigue funcionando. ✅

👆 Haz clic para revelar

REST y HTTP

Los microservicios hablan entre sí usando HTTP. Hay 5 verbos: GET (leer), POST (crear), PUT (actualizar completo), PATCH (actualizar parcial), DELETE (borrar). Los códigos 2xx=éxito, 4xx=error cliente, 5xx=error servidor. ✅

👆 Haz clic para revelar

Arquitectura Hexagonal

Tiene 3 capas: el Dominio (lógica de negocio pura, Java plano), los Puertos (interfaces Java que definen el contrato) y los Adaptadores (@Controller, @Repository, SDKs externos). El Dominio nunca depende del exterior. ✅

Fundamentos

Programación Orientada a Objetos en Java 🧬

Spring Boot es 100% orientado a objetos. Antes de la magia del framework, necesitamos dominar las clases y objetos de Java.

🌎 Mundo Real

Un molde de galletas define la forma. Puedes hacer muchas galletas con ese molde, cada una con su propio relleno (chocolate, vainilla). El molde es la clase, las galletas son los objetos.

💻 En Java

Una Clase define la estructura (campos y métodos). Cada Objeto es una instancia con sus propios valores. La clase Producto puede tener 1000 productos distintos en memoria.

Diagrama de Clase: Producto

Producto

Métodos

Los campos son private para proteger los datos (encapsulación). Los métodos get/set son la puerta de acceso controlada.

En código Java

Producto.java
public class Producto {

  // Atributos (estado del objeto)
  private Long id;
  private String nombre;
  private Double precio;
  private Integer stock;

  // Constructor (crea el objeto)
  public Producto(Long id, String nombre,
                   Double precio, Integer stock) {
    this.id = id;
    this.nombre = nombre;
    this.precio = precio;
    this.stock = stock;
  }

  // Método de negocio
  public boolean estaDisponible() {
    return this.stock > 0;
  }
}

Conceptos Clave

Las Anotaciones de Spring 🏷️

Las anotaciones son instrucciones que le das al framework. Son como etiquetas que Spring lee para saber qué papel cumple cada clase. Haz clic en cada una para explorarla.

@SpringBootApplication

El punto de arranque. Activa todo Spring Boot.

@RestController

Marca una clase como controlador REST que devuelve JSON automáticamente.

@GetMapping

Conecta un método Java con una petición HTTP GET.

@PostMapping

Conecta un método Java con una petición HTTP POST (crear).

@RequestBody

Convierte el JSON del body de la petición en un objeto Java.

@PathVariable

Extrae un valor de la URL. Ej: /productos/{id}

@Service

Marca la clase como servicio de negocio (capa de lógica).

@Autowired

Inyección de dependencias: Spring crea e inyecta el objeto automáticamente.

Live Coding

Tu Primer Endpoint REST 🛰️

Un endpoint es una URL que responde a peticiones HTTP. Vamos a crear nuestro primero: GET /api/productos. Haz clic en cada paso para desbloquear el código.

ProductoController.java
@RestController ← hace que responda JSON
@RequestMapping("/api/productos") ← URL base
public class ProductoController {

  // Inyectamos el servicio (dependencia)
  private final ProductoService service;

  public ProductoController(ProductoService service) {
    this.service = service;
  }

  // GET /api/productos → lista todos
  @GetMapping
  public List<Producto> getAll() {
    return service.findAll();
  }

  // GET /api/productos/{id} → uno solo
  @GetMapping("/{id}")
  public Producto getById(@PathVariable Long id) {
    return service.findById(id);
  }

}
1 Declarar el Controlador

Anotamos la clase con @RestController y definimos la ruta base con @RequestMapping. Spring Boot registra automáticamente esta clase como manejador de peticiones HTTP.

2 Inyectar el Servicio

Usamos inyección por constructor (mejor práctica). Spring crea la instancia de ProductoService y la pasa automáticamente. El controlador nunca sabe cómo se crea el servicio.

3 Endpoint GET (Listar)

@GetMapping sin argumento responde en la ruta base. Retorna una lista de Producto que Spring convierte a JSON automáticamente. El cliente recibe un array JSON.

4 Endpoint GET (Por ID)

@PathVariable extrae el {id} de la URL. Si pides /api/productos/3, el método recibe id=3 como parámetro Long automáticamente.

CRUD

CRUD en Memoria: La Lista como Base de Datos 💾

Antes de conectar una BD real, usamos una lista Java como almacenamiento temporal. Es perfecta para aprender el patrón CRUD sin complejidad adicional.

El Service: La lógica

ProductoService.java
@Service
public class ProductoService {

  // 📋 "Base de datos" en memoria
  private List<Producto> productos = new ArrayList<>();
  private Long nextId = 1L;

  // CREATE
  public Producto save(Producto p) {
    p.setId(nextId++);
    productos.add(p);
    return p;
  }

  // READ ALL
  public List<Producto> findAll() {
    return productos;
  }

  // DELETE
  public void delete(Long id) {
    productos.removeIf(p -> p.getId().equals(id));
  }
}

¿Por qué una lista en memoria?

✅ Ventajas
  • Sin configuración de BD
  • Rápido para prototipar
  • Perfecto para aprender el patrón
  • Los datos se generan solos
⚠️ Limitaciones
  • Se pierden al reiniciar
  • No escala a producción
  • Solo un proceso puede acceder
  • Sin relaciones entre tablas
